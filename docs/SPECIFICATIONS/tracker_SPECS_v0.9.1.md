## 2. 需求详情

### 2.1 REQ-001: CP 页面覆盖率左右布局显示

**需求描述**:
当前 CP 表格中，每一行的覆盖率信息（实际覆盖率 + 关联 TC 数量）是上下分两行显示，导致每行 CP 高度较大。期望改为左右显示，减少行高。

**前端需求**:
- 修改 `.coverage-badge` 和 `.coverage-detail` CSS 样式
- 使用 flexbox 实现左右排列
- 确保不影响移动端显示

**当前代码位置**: `dev/index.html` 第 1354 行

---

### 2.2 REQ-002: 关于对话框增加用户反馈功能

**需求描述**:
在"关于"对话框中增加用户反馈功能，允许用户：
1. 填写反馈条目（类型、描述）
2. 提交反馈后自动生成 JSON 文件
3. 保存到用户数据目录

**前端需求**:
- 在 versionModal 中添加"反馈"标签页
- 反馈表单：类型选择 (Bug/Feature/Optimization) + 描述文本框
- 提交按钮 + 反馈列表展示
- 提交成功提示

**后端需求**:
- 新增 API: `POST /api/feedback`
- 保存位置: `shared/data/user_data/feedbacks/FEEDBACK_{timestamp}.json`
- 权限控制：需登录用户

**数据格式**:
```json
{
  "id": "FEEDBACK_001",
  "type": "bug|feature|optimization",
  "title": "简短描述",
  "description": "详细描述",
  "created_at": "2026-03-07T12:00:00Z",
  "user": "admin"
}
```

---

### 2.3 ISSUE-014: switchTab 函数改进

**需求描述**:
当前 `switchTab` 函数依赖全局 `event` 对象，在某些边缘情况下可能失效。

**前端需求**:
- 修改函数签名：`switchTab(tab, event)`
- 使用 `event.currentTarget` 替代 `event.target`
- 确保非事件调用场景正常工作

---

### 2.4 ISSUE-015: 常量替换 (app_constants.js)

**需求描述**:
`app_constants.js` 文件已创建但代码中未实际使用，仍使用硬编码字符串。

**前端需求**:

- 优先替换关键常量：`SESSION_KEYS.USER`, `API_ENDPOINTS.PROJECTS` 等
- 确保替换后功能正常

## 4. API 接口设计

### 4.1 接口列表

| 方法 | 路径 | 功能 | 状态 |
|------|------|------|------|
| POST | `/api/feedback` | 提交用户反馈 | ⏳ 待实现 |

### 4.2 详细 API 规范

#### 4.2.1 提交用户反馈

**端点**: `POST /api/feedback`

**请求头**:
```
Content-Type: application/json
```

**请求体**:
```json
{
    "type": "bug",
    "title": "页面显示问题",
    "description": "详细描述..."
}
```

**响应**:
```json
{
    "success": true,
    "message": "反馈提交成功",
    "data": {
        "id": "FEEDBACK_001",
        "type": "bug",
        "title": "页面显示问题",
        "created_at": "2026-03-07T12:00:00Z"
    }
}
```

**错误响应**:
```json
{
    "error": "请先登录"
}
```

| 状态码 | 含义 |
|--------|------|
| 200 | 成功 |
| 400 | 请求参数错误 |
| 401 | 未登录 |
| 500 | 服务器错误 |

### 5.2 界面规范

#### 5.2.1 反馈标签页

**表单字段**:
| 字段名 | 类型 | 必填 | 默认值 |
|--------|------|------|--------|
| type | select | Y | bug |
| title | text | Y | - |
| description | textarea | Y | - |

**类型选项**:
- Bug (问题缺陷)
- Feature (功能需求)
- Optimization (优化建议)

**交互逻辑**:
1. 用户选择类型、填写标题和描述
2. 点击"提交反馈"按钮
3. 调用 POST /api/feedback
4. 成功后在列表中显示新反馈
5. 显示成功提示
